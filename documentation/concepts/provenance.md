# Provenance

CAP responses can carry provenance so a client can judge what produced a result.

Provenance is not decoration. It is part of how CAP lets clients calibrate trust in causal claims.

## What Provenance Tells You

Typical provenance fields include:

- `algorithm`
- `graph_version`
- `graph_timestamp`
- `computation_time_ms`
- `mechanism_family_used`
- `mechanism_model_version`
- `server_name`
- `server_version`
- `cap_spec_version`

Some contracts may also include fields such as `sample_size`.

## Why It Matters

Provenance helps a client answer questions such as:

- which algorithm produced this result
- whether the graph is fresh enough for the task
- whether a mechanism-backed path or a lighter method was used
- whether two results came from the same server and graph version

## How Clients Actually Use It

In practice, provenance is most useful in a few repeatable situations:

- when comparing answers from two servers that appear similar on the surface
- when checking whether a result came from the latest graph or an older snapshot
- when investigating why the same query changed over time
- when deciding whether a result was generated by a mechanism-backed path or a lighter fallback

Provenance becomes especially valuable when read together with:

- `reasoning_mode`
- `identification_status`
- `assumptions`

Those fields explain what kind of claim the server is making. Provenance helps explain how that claim was produced in this specific run.

## A Simple Client Habit

When a response matters, log or retain at least:

- the estimate
- `reasoning_mode`
- `assumptions`
- `graph_version`
- `server_version`

That small habit makes later debugging and cross-run comparison much easier.
